# 常见场景

这一节帮助你判断：哪些页面最适合优先接入 `antdv-superform`。

## 场景一：新增 / 编辑 / 查看共用字段

这是最典型的收益场景。

如果一个实体页面同时有：

- 新增页
- 编辑页
- 详情页

并且三者字段大部分重合，那么你可以用一份 schema 复用到 `SuperForm` 和 `SuperDetail`，维护成本会明显下降。

## 场景二：查询表单 + 表格 + 编辑弹窗

很多后台页面都长这样：

- 顶部搜索区
- 中间数据表格
- 行按钮
- 新增 / 编辑弹窗

`SuperTable` 正是为这种页面准备的。它天然支持：

- `searchForm`
- `rowButtons`
- `rowEditor`
- `detail`

## 场景三：有大量重复按钮规则的后台

如果你们项目里经常重复写这些内容：

- 新增按钮样式
- 删除按钮危险态
- 提交按钮文案
- 权限控制逻辑

那么可以考虑把这些沉到全局 `defaultButtons` 与 `buttonRoles` 配置里。

## 场景四：需要扩展业务私有组件

如果基础字段不够用，也不需要推翻现有体系。你可以通过 `registComponent` 注册项目内部组件，再用 `Ext组件名` 的方式接进 schema。

## 什么页面建议暂缓接入

下面这些情况可以先不要上：

- 页面结构高度不稳定，今天改 Tabs，明天改成完全自定义布局
- UI 自由度远高于字段复用价值
- 团队暂时不习惯 schema 驱动的开发方式

## 接入建议

如果你准备在现有项目里落地，建议从这两类页面开始：

1. 字段多、重复度高的表单页
2. 查询表单和列表编辑耦合很强的后台列表页

这样最容易在第一批页面里看到收益。
